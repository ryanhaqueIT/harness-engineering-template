# Harness DAG Dashboard — Design Spec

**Date**: 2026-04-16
**Status**: Draft
**Author**: Ryan Haque + Claude

---

## 1. Purpose

Build a live, interactive DAG dashboard that visualizes the harness engineering pipeline, all 25 validation gates, and agent activity — modeled after Apache Airflow's UI and Archon's execution DAG. The dashboard is a **self-contained HTML file** with zero external dependencies, generated by a Python script using only stdlib.

### Success Criteria

1. User runs `bash scripts/dashboard.sh` and a browser opens within 3 seconds
2. User runs `bash scripts/validate.sh` in another terminal and watches gates light up green/red in real-time
3. The dashboard replicates Airflow's visual language: colored nodes, smoothstep edges, Grid View history, detail panel
4. Zero dependencies — no npm, no CDN, no build step, no internet required
5. Works on macOS, Linux, and Windows (Git Bash)

---

## 2. Architecture

### System Overview

```
validate.sh runs gates sequentially
  └─ After each gate: appends result to .harness/dashboard_state.json
  └─ After all gates: appends full run to .harness/history/runs.json

dashboard.sh
  └─ Generates .harness/dashboard.html (if stale or missing)
  └─ Starts python -m http.server on port 8099
  └─ Opens browser to http://localhost:8099/dashboard.html

dashboard.html (in browser)
  └─ Polls dashboard_state.json every 2 seconds via fetch()
  └─ Re-renders DAG nodes with updated status/colors
  └─ On tab switch to History: reads runs.json for Grid View
```

### Why Self-Contained HTML

- The harness template is zero-dependency by design — no node_modules, no pip install
- A single HTML file can be committed, versioned, and works offline
- dagre layout algorithm is ~30KB minified, embeddable inline
- The polling approach (JSON file + fetch) requires only `python -m http.server`
- No WebSocket, no SSE, no build step

### File Structure

```
scripts/
  harness_dashboard.py      # Generates dashboard.html from gate config + templates
  dashboard.sh              # One-liner: generate + serve + open browser
  dashboard_hooks.sh        # Bash functions: emit_gate_start/end, init/finalize state

.harness/
  dashboard.html            # Generated self-contained HTML (~1000 lines)
  dashboard_state.json      # Live gate status (written by validate.sh per gate)
  dashboard.pid             # PID of running HTTP server (for cleanup)
  history/
    runs.json               # Array of historical run results (appended per run)
```

### 3.2 Header Bar

Replicates Airflow's HeaderCard:

| Element | Content | Style |
|---------|---------|-------|
| Logo | Harness icon (SVG gear/shield) | 24x24px, left-aligned |
| Repo name | From `.git/` or config | Bold, 16px |
| Branch | Current git branch | Gray badge |
| Grade | Letter from scorecard | Large colored badge (green=A+/A, blue=B, yellow=C, red=D/F) |
| Score | "24/31 checks" | Muted text next to grade |
| Ratchet | "↑ improving (42→38)" or "→ stable" | Green/gray/red arrow + text |
| Last run | Timestamp + total duration | Right-aligned, muted |

### 3.3 Node Design (Airflow TaskNode Faithful)

Each gate is rendered as an SVG group:

```svg
<!-- Node: 200px wide, 56px tall -->
<g transform="translate(x, y)">
  <!-- Background rect with state-colored border -->
  <rect width="200" height="56" rx="6"
        fill="#ffffff" stroke="#10b981" stroke-width="2"/>
  
  <!-- Left color stripe (Archon pattern) -->
  <rect width="4" height="56" rx="3 0 0 3" fill="#10b981"/>
  
  <!-- Gate ID -->
  <text x="14" y="22" font-size="11" font-weight="700" fill="#374151">B5</text>
  
  <!-- Gate name -->
  <text x="36" y="22" font-size="11" fill="#374151">Golden Principles</text>
  
  <!-- Status badge (Airflow Badge) -->
  <rect x="160" y="8" width="30" height="18" rx="9" fill="#059669"/>
  <text x="175" y="20" font-size="10" fill="#fff" text-anchor="middle">✓</text>
  
  <!-- Duration + findings -->
  <text x="14" y="42" font-size="10" fill="#9ca3af">0.8s · 3 violations</text>
</g>
```

**State-dependent styling:**

| State | Border | Stripe | Badge BG | Badge Icon | BG Tint |
|-------|--------|--------|----------|------------|---------|
| passed | `#10b981` | `#10b981` | `#059669` | ✓ | `#f0fdf4` |
| running | `#06b6d4` | `#06b6d4` | `#0891b2` | ◌ (animated) | `#ecfeff` |
| failed | `#ef4444` | `#ef4444` | `#dc2626` | ✗ | `#fef2f2` |
| skipped | `#ec4899` | `#ec4899` | `#db2777` | ⏭ | `#fdf2f8` |
| pending | `#a1a1aa` | `#a1a1aa` | `#71717a` | ○ | `#f4f4f5` |
| warning | `#f59e0b` | `#f59e0b` | `#d97706` | ⚠ | `#fffbeb` |

**Selected node**: Border increases to 4px, blue highlight (`#3b82f6`).

**Running node animation**: The spinner icon rotates via CSS `@keyframes spin`. The node gets a subtle pulsing box-shadow glow (Archon's pattern):
```css
@keyframes glow {
  0%, 100% { filter: drop-shadow(0 0 4px rgba(6,182,212,0.3)); }
  50% { filter: drop-shadow(0 0 8px rgba(6,182,212,0.6)); }
}
```

### 3.4 Edge Design (Airflow Smoothstep)

Edges are SVG `<path>` elements with smoothstep curves:

```svg
<path d="M 200 28 C 230 28, 250 84, 280 84"
      stroke="#d1d5db" stroke-width="1.5" fill="none"/>
```

**Edge states:**
- Default: `#d1d5db` (gray), 1.5px
- Target running: `#06b6d4` (cyan), 2px, animated dash (`stroke-dasharray: 8 4`, CSS animation)
- Target failed: `#ef4444` (red), 2px
- Target passed: `#10b981` (green), 1.5px

**Arrow markers**: Small triangles at edge endpoints pointing in flow direction.

### 3.5 The Four Tab Views

#### Tab 1: Pipeline View (Left-to-Right DAG)

The high-level development flow:

```
[Spec] → [Plan] → [Features] → [Implement] → [Validate] → [Ship]
```

6 nodes, left-to-right. Each node is larger (240x72px) and shows:
- Phase name (bold)
- Status badge
- Key metric (e.g., "12/87 features", "24/25 gates passed")
- Duration

The [Validate] node is expandable — clicking it switches to the Gates tab.

#### Tab 2: Gates View (Top-to-Bottom DAG, 7 Layers)

The 25 validation gates organized by layer:

```
Layer 1 (Deterministic):     [B1 Lint] → [B2 Format]
                                  ↓            ↓
Layer 2 (Structural):   [B4 Imports] [B5 Golden] [B6 Arch] [B8 Wiring]
                              ↓          ↓          ↓          ↓
Layer 3 (Tests):                    [B3 Tests]
                                        ↓
Layer 4 (Build):         [F4 Build] [F6 Smoke] [F7 Playwright]
                              ↓          ↓          ↓
Layer 5 (Infra):              [I1 TfFmt] [I2 TfValidate]
                                   ↓          ↓
Layer 6 (Observability):        [O1 Observability]
                                        ↓
Layer 7 (PRD):           [X5 Features] [X6 Live] [X7 Spec] [R1 Ratchet]
```

Layout uses dagre with `rankdir: 'TB'`, `ranksep: 60`, `nodesep: 30`.

Gates within the same layer have no edges between them (they're parallel). Edges flow from each layer to the next layer's gates. The actual dependency isn't strict (validate.sh runs them sequentially), but the visual communicates the logical progression.

#### Tab 3: Agents View (Left-to-Right Pipeline)

The 8 agent roles as a pipeline:

```
[Planner] → [Tester] → [Executor] → [Build-Fixer] → [Reviewer]
                                          ↑
                                    [Entropy-Cleaner]
                                    
[Bootstrapper] (standalone)
[Post-Build-Reviewer] (standalone, read-only)
```

Each agent node shows:
- Agent name
- Status (active/idle)
- Current task (if active)
- Last active timestamp

Agent status comes from `.harness/dashboard_state.json` `agents` section, updated by hooks.

#### Tab 4: History View (Airflow Grid View)

Exact replication of Airflow's Grid View:

**Y-axis (left column)**: Gate names, grouped by layer with collapsible headers
**X-axis (top row)**: Validation runs, newest-right, each column 18px wide
**Cells**: 14x14px colored badges with state icons
**Duration bars**: At top, proportional to total run duration, colored by overall status
**Hover**: Row + column crosshair highlight

### 3.6 Detail Panel (Airflow Right Panel)

When a gate node is clicked, a right panel slides in (30% width):

**Header card:**
- Gate icon + ID + full name
- Status badge (large, colored)
- Duration
- Findings count

**Tabs:**
1. **Findings**: List of violations
   - Each violation shows: file path, line number, description, fix suggestion
   - Clickable file:line references
2. **Output**: Raw gate script stdout/stderr
3. **History**: Sparkline showing pass/fail trend for this specific gate over last 20 runs

**Close button**: X in top-right returns to full-width DAG view.

### 3.7 Dark Mode

The dashboard supports both light and dark themes, toggled via a button in the header. Dark mode uses Airflow's dark palette:

| Element | Light | Dark |
|---------|-------|------|
| Background | `#ffffff` | `#1a1a2e` |
| Surface | `#f9fafb` | `#16213e` |
| Text primary | `#111827` | `#e5e7eb` |
| Text muted | `#6b7280` | `#9ca3af` |
| Border | `#e5e7eb` | `#374151` |
| Node bg | `#ffffff` | `#1e293b` |

State colors remain the same in both modes (green, red, cyan, etc.) — they're designed to work on both backgrounds.

### 4.3 How validate.sh Writes State

A bash function added to validate.sh:

```bash
init_dashboard() {
  # Write initial state with all gates as "pending"
  python3 -c "
import json, datetime
state = {'version': 1, 'timestamp': datetime.datetime.utcnow().isoformat()+'Z', ...}
json.dump(state, open('.harness/dashboard_state.json','w'), indent=2)
"
}

update_gate() {
  local gate_id="$1" status="$2" duration="$3" findings="$4"
  python3 -c "
import json, sys
state = json.load(open('.harness/dashboard_state.json'))
state['gates']['$gate_id']['status'] = '$status'
state['gates']['$gate_id']['duration_ms'] = $duration
state['gates']['$gate_id']['findings'] = $findings
state['timestamp'] = __import__('datetime').datetime.utcnow().isoformat()+'Z'
json.dump(state, open('.harness/dashboard_state.json','w'), indent=2)
"
}
```

Called as:
```bash
init_dashboard
start_time=$(python3 -c "import time; print(int(time.time()*1000))")
check "B1" "Lint" "ruff check backend/"
end_time=$(python3 -c "import time; print(int(time.time()*1000))")
duration=$((end_time - start_time))
update_gate "B1" "$LAST_STATUS" "$duration" "$LAST_FINDINGS"
```

Note: Uses `python3 -c "import time; print(int(time.time()*1000))"` instead of `date +%s%N` for cross-platform compatibility (macOS lacks nanosecond support in `date`).

### 4.4 How dashboard.html Polls

```javascript
const POLL_INTERVAL = 2000; // 2 seconds
let currentState = null;

async function poll() {
  try {
    const resp = await fetch('dashboard_state.json?t=' + Date.now());
    const newState = await resp.json();
    if (JSON.stringify(newState) !== JSON.stringify(currentState)) {
      currentState = newState;
      renderHeader(currentState);
      renderDAG(currentState.gates);
      renderStatusBar(currentState);
    }
  } catch (e) {
    // File not ready yet, retry
  }
}

setInterval(poll, POLL_INTERVAL);
poll(); // Initial load
```

Cache-bust via `?t=timestamp` query parameter prevents stale reads.

## 5. Technical Implementation

### 5.1 DAG Layout (dagre inline)

The dagre library (~30KB minified) is embedded directly in the HTML as an inline `<script>` block. Layout is computed once on load and recomputed only when the tab changes.

```javascript
function layoutGates(gates) {
  const g = new dagre.graphlib.Graph();
  g.setGraph({ rankdir: 'TB', ranksep: 60, nodesep: 30, marginx: 20, marginy: 20 });
  g.setDefaultEdgeLabel(() => ({}));
  
  // Add nodes
  for (const [id, gate] of Object.entries(gates)) {
    g.setNode(id, { width: 200, height: 56, ...gate });
  }
  
  // Add edges (layer N → layer N+1)
  // Layer 1 gates → Layer 2 gates, etc.
  addLayerEdges(g, gates);
  
  dagre.layout(g);
  return g;
}
```

### 5.2 SVG Rendering

All nodes and edges are rendered as SVG elements inside a single `<svg>` with viewBox for responsive scaling:

```html
<svg id="dag-canvas" viewBox="0 0 {width} {height}" 
     style="width:100%; height:100%;">
  <defs>
    <marker id="arrow" .../>  <!-- Edge arrow markers -->
  </defs>
  <g id="edges">...</g>       <!-- Edge paths (below nodes) -->
  <g id="nodes">...</g>       <!-- Node groups (above edges) -->
</svg>
```

Pan and zoom via CSS `transform` on the root `<g>`, controlled by mouse wheel (zoom) and drag (pan).

### 5.3 Interaction

- **Click node**: Sets `selectedNode`, re-renders with 4px blue border, opens detail panel
- **Click background**: Deselects node, closes detail panel
- **Hover node**: Shows tooltip with gate name, status, duration
- **Hover Grid View cell**: Highlights row + column
- **Click tab**: Switches view, re-layouts if needed
- **Click theme toggle**: Switches light/dark, saves preference in localStorage

### 5.4 Responsive Behavior

- **> 1200px**: Full layout (DAG + detail panel side by side)
- **800-1200px**: Detail panel overlays DAG (modal-style)
- **< 800px**: Tabs stack vertically, nodes shrink to 160px width

## 6. Scripts

### 6.1 `scripts/harness_dashboard.py`

**Purpose**: Generates `.harness/dashboard.html`

**Behavior**:
1. Reads gate configuration from validate.sh (parse gate IDs and names)
2. Reads current state from `.harness/dashboard_state.json` (if exists)
3. Reads run history from `.harness/history/runs.json` (if exists)
4. Generates a self-contained HTML file with:
   - Inline CSS (~250 lines)
   - Inline JS (~500 lines including dagre minified)
   - SVG templates for nodes, edges, grid cells
   - The gate DAG structure as a JS constant
5. Writes to `.harness/dashboard.html`

**Usage**:
```bash
python3 scripts/harness_dashboard.py                    # Generate HTML
python3 scripts/harness_dashboard.py --open             # Generate + open in browser
python3 scripts/harness_dashboard.py --serve             # Generate + start HTTP server
python3 scripts/harness_dashboard.py --serve --open      # All three
```

**Dependencies**: Python 3.7+ stdlib only (json, pathlib, subprocess, http.server, webbrowser)

### 6.2 `scripts/dashboard.sh`

**Purpose**: One-command dashboard launch

**Behavior**:
```bash
#!/usr/bin/env bash
set -euo pipefail

HARNESS_DIR="$(cd "$(dirname "$0")/.." && pwd)/.harness"

# Generate dashboard HTML
python3 "$(dirname "$0")/harness_dashboard.py"

# Start HTTP server in background (kill existing if running)
if [ -f "$HARNESS_DIR/dashboard.pid" ]; then
  kill "$(cat "$HARNESS_DIR/dashboard.pid")" 2>/dev/null || true
fi

cd "$HARNESS_DIR"
python3 -m http.server 8099 &>/dev/null &
echo $! > dashboard.pid

# Open browser
if command -v open &>/dev/null; then
  open "http://localhost:8099/dashboard.html"
elif command -v xdg-open &>/dev/null; then
  xdg-open "http://localhost:8099/dashboard.html"
elif command -v start &>/dev/null; then
  start "http://localhost:8099/dashboard.html"
fi

echo "Dashboard running at http://localhost:8099/dashboard.html"
echo "PID: $(cat dashboard.pid)"
```

### 6.3 Modifications to `scripts/validate.sh`

Add at the top:
```bash
# Dashboard state management
DASHBOARD_STATE=".harness/dashboard_state.json"
source "$(dirname "$0")/dashboard_hooks.sh" 2>/dev/null || true
```

Wrap each `check()` call to emit state updates:
```bash
# Before: check "B1" "Lint" "ruff check backend/"
# After:
emit_gate_start "B1"
check "B1" "Lint" "ruff check backend/"
emit_gate_end "B1" "$?" "$GATE_DURATION" "$GATE_FINDINGS"
```

The `dashboard_hooks.sh` script contains `emit_gate_start`, `emit_gate_end`, `init_dashboard_state`, and `finalize_dashboard_run` functions. If the file doesn't exist, validate.sh works exactly as before (the `|| true` makes it a no-op).

## 7. Scope Boundaries

### In Scope (v1)

- Self-contained HTML dashboard with 4 tabs (Pipeline, Gates, Agents, History)
- Airflow-faithful visual design (colors, nodes, edges, Grid View, detail panel)
- Live polling from JSON state file (2-second interval)
- Dark/light theme toggle
- `harness_dashboard.py` generator script
- `dashboard.sh` launcher script
- `dashboard_hooks.sh` for validate.sh integration
- History persistence (runs.json, max 100 runs)
- Pan/zoom on DAG canvas
- Click-to-select nodes with detail panel
- Responsive layout (desktop + tablet)

### Out of Scope (v2+)

- Drag-and-drop DAG editing (Archon's builder mode)
- WebSocket/SSE real-time updates (polling is sufficient)
- Gantt chart view (gates run too fast, <2s each)
- Authentication or multi-user support
- Deployment to remote servers
- npm/React/build step of any kind
- Integration with external monitoring (Grafana, DataDog)
- Mobile-optimized layout
- Animated node transitions (CSS transitions for color changes are in scope)

## 8. Testing Plan

1. **Generate test**: Run `python3 scripts/harness_dashboard.py`, verify HTML is valid and opens in browser
2. **Polling test**: Start dashboard, run validate.sh, confirm nodes update in real-time
3. **History test**: Run validate.sh 3 times, confirm Grid View shows 3 columns
4. **Detail panel test**: Click a failed gate, confirm violations are shown
5. **Dark mode test**: Toggle theme, confirm all elements are readable
6. **Empty state test**: Run dashboard with no state file, confirm graceful "waiting for first run" message
7. **Windows test**: Run on Git Bash, confirm no `grep -oP` or bash-ism failures
8. **Cross-browser test**: Chrome + Firefox at minimum
